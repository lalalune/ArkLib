# ArkLib-to-Whole-zkVM Verification Boundary

ArkLib is a proof-system verification library. It can be a necessary part of a
verified zkVM stack, but it is not by itself a proof that a concrete zkVM
verifier accepts only correct VM executions.

This page records the final boundary theorem ArkLib is meant to support and
names the assumptions that must be supplied by ArkLib or by adjacent projects.
The matching Lean template is
`ArkLib/ProofSystem/ZkVMBoundary.lean`.

## Final Claim Shape

The whole-zkVM claim should have this shape:

```text
If a concrete verifier accepts a concrete proof for public input x,
and that verifier/proof execution is equivalent to the ArkLib protocol spec,
and the ArkLib proof-system theorem extracts or validates an arithmetization witness,
and the arithmetization witness exactly encodes a valid VM trace,
then there exists a valid VM execution trace producing the claimed public output.
```

The Lean template names this as `WholeZkVMEndToEndClaim`.

## ArkLib Scope

ArkLib can own these parts of the final theorem:

- Interactive oracle reductions, protocol specs, and composition/lifting APIs.
- Proof-system soundness/completeness/knowledge-soundness for Sumcheck, FRI,
  STIR, WHIR, Binius, LogUp, RingSwitching, Spartan, Plonk, and related protocols.
- Commitment and oracle-query abstractions used by proof-system theorems.
- Fiat-Shamir and BCS transforms at the mathematical protocol/spec level.
- Coding-theory, proximity, list-decoding, and polynomial facts used by those protocols.
- Named residual APIs for trusted external paper assumptions that are not yet
  internally proved.

## External Boundary Items

A whole-zkVM project must supply these interfaces outside the ArkLib proof-system
core:

- VM instruction semantics, such as RISC-V or EVM semantics, and the public
  output convention.
- Trace model: step relation, registers, memory, nondeterminism, halting, and
  failure states.
- Arithmetization correctness: constraints exactly encode valid traces,
  lookups, range checks, memory consistency, and permutation arguments.
- Protocol binding: the ArkLib relation consumed by the proof system is exactly
  the arithmetized trace relation.
- Fiat-Shamir, BCS, commitment, transcript, random-oracle, and query-opening
  implementation equivalence to ArkLib specs.
- Concrete prover/verifier implementation equivalence to ArkLib executable specs
  or mathematical specs.
- Serialization and deployment boundary, including on-chain verifier correctness
  if deployed verification is part of the claim.
- Recursion and aggregation soundness when the zkVM uses recursive proofs.

## Named Guest-Program Provider

For the current roadmap, [`Verified-zkEVM/evm-asm`](https://github.com/Verified-zkEVM/evm-asm)
is the named provider for the guest-program and VM-semantics side of this boundary.
It should discharge the RV64/EVM execution semantics, bounded guest-program
correctness, public input/output, halt conventions, and zkVM codegen/integration
contract. ArkLib should not re-prove those VM facts; it should consume them
through a relation such as `GuestTraceRelation` or `ArithmetizedTraceRelation`.

The remaining cross-repo bridge is explicit: prove that the relation supplied to
ArkLib by the arithmetization layer is exactly the relation generated by the
`evm-asm` guest trace and public I/O boundary.

## Lean Interfaces

`WholeZkVMInterfaces` names the abstract objects crossing the boundary:

- `PublicInput`
- `ConcreteProof`
- `ConcreteVerifier`
- `ArkStatement`
- `ArkProof`
- `VmTrace`
- `ArithmetizationWitness`
- `RecursionArtifact`
- `DeploymentArtifact`

`WholeZkVMAssumptions` names the proof obligations:

- `concreteVerifierAccepts`
- `implementationMatchesArkSpec`
- `arkProofSystemSoundness`
- `arithmetizationSoundness`
- `vmExecutionValid`
- `transcriptAndCommitmentBinding`
- `recursionSoundness`
- `deploymentSoundness`

The theorem template is deliberately abstract. A concrete zkVM should instantiate
these fields with real types and replace each assumption field with a theorem.

## Milestone Mapping

Existing ArkLib milestone issues should be read as discharging only parts of
`WholeZkVMAssumptions`:

| Assumption | ArkLib-facing work |
|---|---|
| `arkProofSystemSoundness` | Soundness/completeness/knowledge-soundness for Sumcheck, FRI, STIR, WHIR, Binius, LogUp, RingSwitching, Spartan, Plonk, and related composition lemmas. |
| `transcriptAndCommitmentBinding` | BCS, Fiat-Shamir, commitment, Merkle, KZG, random-oracle, query-opening, and transcript-binding work. |
| `implementationMatchesArkSpec` | Future executable-spec / extracted-code / hand-written-verifier equivalence work. |
| `arithmetizationSoundness` | Constraint-system and protocol-binding work, plus external arithmetization repos. |
| `vmExecutionValid` | External VM semantics projects. |
| `deploymentSoundness` | External serialization and on-chain verifier projects. |
| `recursionSoundness` | Recursive-proof and aggregation projects when used. |

## Non-Claim

Finishing ArkLib is necessary but not sufficient for total zkVM verification.
Even a fully green ArkLib proof-system library would still need the VM,
arithmetization, implementation-equivalence, deployment, and recursion links
above before anyone can claim an end-to-end verified zkVM.
